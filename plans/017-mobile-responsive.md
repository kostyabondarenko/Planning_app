# План: Адаптация UI под iPhone (мобильная вёрстка)

## Общее описание

Адаптировать текущий dashboard, страницу цели, kanban «Ближайшие дни» и календарь под экраны iPhone (~390×844px). Важное условие: **сначала проверить текущий UI на реальном iPhone** и адаптировать только те места, которые реально ломаются или неудобны.

Этот план выполняется **после** планов 014 (PWA) и 015 (деплой) — когда есть рабочая установка на телефоне для ревью.

---

## Входящие требования

1. Все страницы читаются и функциональны на iPhone SE (375px) и iPhone 15 (393px)
2. Touch-targets ≥ 44×44px (требование Apple HIG)
3. Нет горизонтальной прокрутки
4. Учтена safe-area (notch сверху, home indicator снизу)
5. Kanban-доска: либо горизонтальная прокрутка колонок, либо одна колонка с табами
6. Таблицы/широкие списки — адаптированы (стеком, а не колонками)
7. Модалки — fullscreen на мобильном

---

## Промпт для запуска плана

```
Выполни план plans/017-mobile-responsive.md.

Используй скиллы: brandbook-stylist, react-forms

ВАЖНО: план начинается с этапа аудита. Не адаптируй страницы

iPhone (или по скриншотам от пользователя) и составь список проблем. 
Дождись подтверждения списка перед рефакторингом.
```

## План реализации

### Этап 1: Аудит текущего UI на iPhone

**Выполняется вручную + Chrome DevTools:**

1. Открыть приложение на реальном iPhone (через план 015)
2. Пройти по всем страницам и сделать скриншоты:
   - `/login`, `/register`
   - `/dashboard` (список целей)
   - `/dashboard/goal/[id]` (страница цели с вехами)
   - `/dashboard/daily` (kanban «Ближайшие дни»)
   - `/dashboard/calendar`
   - Модалки: создание цели, редактирование действия, редактирование дат задачи
3. Составить таблицу **«Страница → Проблема → Приоритет»**:
   - P1 — ломает функциональность (не нажать, не прочитать)
   - P2 — неудобно, но работает
   - P3 — косметика

**Результат этапа:** файл `plans/017-audit-report.md` с таблицей проблем и скриншотами.

### Этап 2: Глобальные утилиты

Базовые вещи, нужные везде:

**`frontend/src/app/globals.css`:**
```css
/* Safe-area уже в плане 014, здесь — дополнительно */
.pb-safe { padding-bottom: max(1rem, var(--safe-bottom)); }
.pt-safe { padding-top: max(1rem, var(--safe-top)); }

/* Предотвратить zoom на iOS при фокусе input (требует font-size ≥ 16px) */
input, select, textarea { font-size: 16px; }

/* Tap-highlight убрать */
* { -webkit-tap-highlight-color: transparent; }

/* Touch-targets */
.tap-target { min-height: 44px; min-width: 44px; }
```

**Tailwind breakpoints** (проверить `tailwind.config.ts`):
- `sm: 640px` — базовый mobile breakpoint (будем считать <640 = mobile)
- Предпочитать mobile-first: базовые стили для mobile, `sm:` / `md:` — для desktop

### Этап 3: Навигация

**Проблема:** sidebar с профилем (из плана 013 этап 6) занимает место на мобильном.

**Решение:**
- На десктопе (sm+): sidebar как сейчас
- На мобильном: **bottom tab bar** с 4 иконками (Цели, День, Календарь, Профиль) — стандартный iOS-паттерн
- Учесть `var(--safe-bottom)` для home indicator

### Этап 4: Dashboard (список целей)

Обычно уже работает — проверить:
- Карточки целей: одна колонка на мобильном (grid-cols-1)
- Кнопка «+ Цель»: FAB (floating action button) справа снизу, над bottom nav

### Этап 5: Страница цели с вехами

**Вероятные проблемы:**
- Таблица действий с колонками «Название / Период / Прогресс / Статус» — не влезет
- Решение: карточки-стеки на мобильном

**Шаблон карточки действия:**
```tsx
<div className="sm:hidden p-4 rounded-xl bg-app-card border">
  <div className="font-semibold">{action.title}</div>
  <div className="text-sm text-app-textMuted">{period}</div>
  <ProgressBar value={action.current_percent} />
  <div className="flex justify-between mt-2">
    <Badge>{action.status}</Badge>
    <button className="tap-target">...</button>
  </div>
</div>
<table className="hidden sm:table">...</table>
```

### Этап 6: Kanban «Ближайшие дни»

**Проблема:** 7 колонок (дней) по ширине не влезут.

**Варианты:**
- **A) Горизонтальная прокрутка** — `overflow-x-auto snap-x snap-mandatory`, одна колонка на экран со snap. Свайп между днями.
- **B) Табы** — сверху 7 коротких кнопок (Пн/Вт/…), ниже одна колонка выбранного дня.

**Рекомендация:** B (табы) — привычнее, меньше жестов. dnd-kit drag&drop сохраняется внутри одной колонки; для переноса между днями — долгий тап или отдельная кнопка «Переместить».

Важно: **проверить, что dnd-kit работает с touch-событиями.** Возможно потребуется `TouchSensor` с `activationConstraint: { delay: 250, tolerance: 5 }` чтобы не конфликтовало со скроллом.

### Этап 7: Календарь

- Месячная сетка 7×6 — нормально влезает, ячейки узкие
- При тапе на день — bottom sheet со списком задач этого дня (вместо модалки по центру)
- `DeadlineTasksList` — карточки стеком

### Этап 8: Модалки

Все модалки на мобильном — **fullscreen sheet** (сверху вниз):
```tsx
<div className={clsx(
  "fixed z-50",
  // Mobile: fullscreen bottom sheet
  "inset-x-0 bottom-0 top-[10vh] rounded-t-3xl",
  // Desktop: центр
  "sm:inset-auto sm:top-1/2 sm:left-1/2 sm:-translate-x-1/2 sm:-translate-y-1/2 sm:rounded-2xl sm:max-w-md"
)}>
```

Focus trap и закрытие по Escape — уже есть в скилле `react-forms`.

### Этап 9: Формы и инпуты

- `font-size: 16px` везде (иначе iOS зумит)
- Date-picker: использовать нативный `<input type="date">` на мобильном
- Нативная клавиатура с подсказками: `inputMode="numeric"` для чисел

### Этап 10: Финальный чеклист на реальном iPhone

Пройти все сценарии:
- [ ] Создать цель, добавить веху, добавить действие
- [ ] Залогировать выполнение регулярного действия
- [ ] Переместить задачу между днями в kanban
- [ ] Открыть календарь, тапнуть день, отредактировать дату задачи
- [ ] Сменить light/dark тему
- [ ] Safe-area не перекрывает контент
- [ ] Нет горизонтальной прокрутки нигде
- [ ] Все кнопки/чекбоксы ≥ 44px
